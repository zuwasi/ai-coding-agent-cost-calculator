An explenation what this page does

This page is a planning model for the risk described in “Microsoft Couldn’t Afford Claude Code. That Should Terrify Every Engineering Team.” The article argues that the danger is not that coding agents fail; the danger is that successful adoption turns AI coding into a variable, token-metered compute bill.

Core article claim

The article claims Microsoft rolled Claude Code out broadly, saw heavy adoption, and later decided to shut off licenses because usage was “very popular, perhaps a little too popular.” It also cites an Uber-scale example of 5,000 engineers and very high AI budget burn. Treat those numbers as article claims and scenario prompts, not as audited vendor billing data.

What the calculator models

  • Adoption: how many engineers actually use the tool.
  • Frequency: how many sessions each active engineer runs per day.
  • Token load: how much context and output each session consumes.
  • Model price: current public pricing or documented multipliers where available.
  • Agent behavior: retries, tool calls, validation loops, background workers, and multi-agent orchestration.

Agent-specific sources

  • Amp manual: deep/smart/rush modes, subagents, Oracle/Librarian, and pass-through pricing assumptions.
  • Claude Code docs: terminal/IDE/web agent behavior, model aliases, subscription/API caveats, and token billing context.
  • Cursor docs: Agent workflow, Auto + Composer pool, API pool, Max Mode, and model pricing caveats.
  • Factory Droid docs: CLI/App/Exec, Missions, AGENTS.md, skills, hooks, MCP, Droid Core, model multipliers, and rolling rate limits.

How to read the result

A high result does not prove a tool is bad. It shows where governance is needed: model selection, context discipline, caching, budgets, rate limits, review policies, and a rollout plan. The safest interpretation is to use this calculator before procurement, pilot expansion, or organization-wide deployment.

FreeRTOS → Loongson ISO 26262 RTOS flowgraph

This use case is a safety-critical migration program: move a FreeRTOS-based embedded stack to Loongson CPUs, then harden it into an ISO 26262-oriented RTOS delivery where production C code must pass MISRA C compliance gates. The central engineering issue is not only “can FreeRTOS run on Loongson?” but whether the complete hardware/software system can demonstrate safety mechanisms, spatial independence, temporal independence, controlled failure modes, and enough evidence for an assessor. The AI-agent cost risk is high because each task needs architecture context, hardware-port knowledge, standards evidence, static-analysis cleanup, and repeated verification loops.

1. Safety scopeDefine item boundary, ASIL target, safety-related vs non-safety software, Loongson target, toolchain, and certification strategy.
2. Independence conceptDesign spatial independence with MMU/MPU-style protection and temporal independence with timing budgets, monitors, and watchdog strategy.
3. Porting gapKernel port layer, LoongArch context switch, interrupt model, timers, memory model, boot flow, BSP, drivers, and compiler/runtime deltas.
↓ validation and evidence gates ↓
4. Kernel hardeningPartition safety tasks, isolate non-safety tasks, control privileged access, define scheduler assumptions, and handle common-cause failure risk.
5. MISRA C 2025 cleanupStatic analysis, rule deviations, code rewrites, generated evidence, peer-review packages, and enforceable CI gates.
6. Safety caseTraceability, requirements, hazard links, verification matrix, coverage targets, tool qualification notes, and release artifacts.
↓ release decision ↓
7. Hardware verificationBoard bring-up, timing tests, interrupt latency, stress, fault injection, watchdog-window tests, and regression automation.
8. Assessment packageSource, docs, safety manual, test reports, MISRA deviations, SBOM, reproducible build, and integration guide.
9. Maintenance loopSecurity fixes, Loongson variants, regression suites, compliance delta analysis, customer support, and recertification impact checks.

Why this matters for the calculator: agent sessions are not simple chat. They include code search, Loongson/LoongArch porting decisions, safety architecture tradeoffs, standards interpretation, MISRA repair, timing-analysis support, test generation, evidence writing, and re-checking. That is why the preset uses high context and loop multipliers even with only 50 engineers.
